本项目为 [openclaw](https://github.com/openclaw/openclaw) 最小实践，将按照以下需求逐步开发。P0 范围：仅 CLI 配置与本地对话；Telegram 真实对话、定时任务与 MCP 的具体实现属后续迭代。

以下所有命令行交互，命令行格式均为类似 `tclaw <command>` 的形式；

P0

1. 命令行交互

1.1 可通过命令行进行配置，命令
`tclaw config`
进入配置，用户可以选择配置的内容（单选）；

1.1.1 可通过命令行配置 TELEGRAM BOT 的相关配置，例如TELEGRAM_BOT_TOKEN，仅支持Telegram 【P0 暂时不需要实现真实的 Telegram 对话】；

1.1.2 可通过命令行配置 OPENAI 的相关配置，例如OPENAI_API_KEY，仅支持 openai api 模型；

命令行配置每次输入一项，例如

```bash
tclaw config
Choose the configuration you want to configure:
[1] TELEGRAM BOT
[2] OPENAI
Enter the number of the configuration you want to configure:
```

```bash
Choose the openai configuration you want to configure:
[1] ALL
[2] API KEY (Required)
[3] BASE URL
[4] MODEL NAME
Enter the number of the configuration you want to configure:
```

如果已经配置好的项，要在末尾加上 `[Configured]` 标记。如果是ALL，那么按顺序一个一个配置；

```bash
TELEGRAM_BOT_TOKEN, Enter the value
```

1.1.3 可通过命令行查看当前的所有配置（列出所有配置项 key 及脱敏后的值）

```
tclaw config list
```

1.2 可通过命令行进行对话，命令
`tclaw chat`
进入对话界面，用户可以输入对话内容；

1.2.1 通过对话 `/exit` 退出对话；

1.2.2 通过对话 `/summarize` 总结对话内容，并保存到日志文件中；

1.2.3 通过对话 `/clear` 清空所有历史与记忆（chat.md 与 memory.md），需二次确认（输入 yes 或 y）后执行，相当于开启全新对话；

1.2.4 用户退出对话后，在既有记忆摘要基础上增量合并本轮对话，生成新摘要并写入 `~/.tclaw/memory.md`；

1.2.5 用户重启对话后，读取 `~/.tclaw/chat.md` 显示对话内容；展示时以上下文形式仅加载 `~/.tclaw/memory.md` 中的记忆摘要，完整历史仍存于 chat.md；调用 LLM 时显示「Thinking」等进度反馈，避免用户误以为程序卡顿；

2. 配置

2.1 用户的所有配置存储在 `~/.tclaw` 目录下；首次使用时自动创建该目录及下述文件。

2.2 用户的历史对话均存储在 `~/.tclaw/chat.md` 文件中；

2.3 用户的对话历史记忆（总结后，每次对话都会重新调用）均存储在 `~/.tclaw/memory.md` 文件中；

2.4 agent 学习的 skill 存储在 `~/.tclaw/skills/*.md` 目录中，每个 skill 为一个独立的 Markdown 文件；

2.5 agent 的人格定义存储在 `~/.tclaw/personality.md` 文件中；

2.6 用户的配置（例如 TELEGRAM_BOT_TOKEN, OPENAI_API_KEY）均存储在 `~/.tclaw/config.json` 文件中；

P1

1. 命令行执行

1.1 用户可以通过对话(chat)执行以下行为，例如：

1.1.1 设定定时任务（具体规范待补充；接口与调用链在架构文档中预留）。

1.1.2 调用 MCP（P0 仅列出，具体规范待补充；接口与调用链在架构文档中预留）。

1.1.3 执行系统操作

    输入：用户以自然语言表达需求，例如
    ```bash
    > 帮我看看桌面有什么文件
    ```

    行为：由 LLM 理解意图并决定要执行的系统命令（如 `ls -l ~/Desktop`），经安全策略校验后执行，将 stdout/stderr 与退出码回传 LLM，由 LLM 生成面向用户的自然语言回复。

    输出：用户得到基于执行结果的整理后回复（例如桌面文件列表的简要说明）。

    边界与安全：
    - 执行超时（如 30 秒）与输出长度截断（如 4KB），避免长时间阻塞或 token 溢出；
    - 禁止执行危险命令（如包含 `sudo`、`rm -rf`、`mkfs`、重定向到块设备等），具体策略见架构与实现。

2. 加载skill

2.1 用户可以在`~/.tclaw/skills/*.md`文件夹中存储skill，skill包含name和description，以

```markdown
---
name: tclaw-feature-from-doc
description: 按「参考 openclaw → 更新文档 → Plan → 依赖判断 → 代码实现 → 测试」顺序，从设计文档片段实现 mini-openclaw 功能。Use when user says /tclaw-feature-from-doc or provides @设计文档.md (xx-xx) and asks to implement that feature.
---
```

开头；frontmatter 之后为该 skill 的详细说明正文。

默认情况下，agent 会在进入 `tclaw chat` 时扫描 `~/.tclaw/skills/*.md` 目录，将所有 skill 的 name 和 description 以及正文要点加载到对话的 system 上下文中，并且自主判断什么时候需要运行什么 skill。当用户提出的需求在语义上明显匹配某个 skill 的 name 或 description 时，应优先遵循该 skill 文档中的约束与说明。

例如，假设有 skill 学猫叫，其文档内容为

```markdown
---
name: kitty miao
description: 打印猫叫的文字
---

# 学猫叫

仅输出`喵呜`即可
```

当用户在对话中，提出要agent学猫叫或类似的说法时，agent应当参考skill并最终输出`喵呜`；

为了做到这一点，需要提前写好prompt让agent知道具体的触发流程；
