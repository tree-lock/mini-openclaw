# mini-openclaw：从设计文档片段实现功能

## 使用场景

当用户在本仓库中：

- 通过 `/create-skill`、`create-skill` 或口头说明「根据这段设计文档实现功能」之类的语句，
- 并给出一段设计文档/架构文档的片段（例如 `@设计文档.md (76-96)`、`@docs/设计文档.md (76-96)`），

你需要把这段文档当成「新需求规格」，并按照下面**固定顺序**完成落地：

1. 参考上游 `openclaw` 的实现方式（只作参考，不强行一比一复制）。
2. 更新/补充本项目 `docs/*.md` 中的相关内容（先文档，后代码）。
3. 进行 Plan 级别的架构规划（必要时切换到 Plan 模式）。
4. 判断是否需要新增或调整依赖。
5. 完成代码实现（遵守本项目分层与约束）。
6. 运行测试与质量检查（test + lint + format）。

## 总体流程

每次被本 Skill 触发时，遵循下面的步骤和顺序，不要跳步，也不要并行执行这些阶段：

### 1. 明确需求与范围

1. 从用户提供的设计文档/架构文档片段中，提炼出：
   - **要实现的具体行为**（输入、输出、边界）；
   - **优先级**（P0 / P1 / 加分项），如果片段中有标注则遵循之；
   - **是否会影响 CLI 接口**（例如新增子命令、对话内新指令）。
2. 用 2–4 句中文总结你对需求的理解，并在回答中显式列出：
   - **目标**（要让 tclaw 多出什么能力）；
   - **受影响的模块**（例如 `config` / `chat` / `memory` / `storage` / `llm` / 新的命令执行模块等）。
3. 如果需求明显属于「后续迭代/加分项」，要在总结中标记出来，但仍然按流程继续实现，除非用户明确说只做设计不实现。

### 2. 参考上游 openclaw 的实现方式

1. 使用你已有的 `openclaw` 仓库知识（例如命令执行、工具调用、定时任务、MCP 等模块）来寻找**相似的功能形态**：
   - 命令/工具是如何抽象的；
   - 配置、会话、权限、安全边界是如何处理的；
   - 是否有值得借鉴的错误处理与日志策略。
2. 在回答中用**简短 bullet 列表**写出你从上游抽象出的 2–5 点设计原则，例如：
   - 工具/命令都走统一适配层；
   - 会话上下文与长期记忆的分离方式；
   - 如何防止随意执行危险系统命令等。
3. 仅作为**设计参考**使用，不要复制上游的具体实现代码，也不要引入与本项目目标明显不匹配的大而全架构。

### 3. 优先更新本地文档（设计文档 & 架构文档 & 依赖文档）

**重要：遵守本仓库规则——先文档，后实现。**

1. 读取以下文档（至少查阅与当前需求最相关的部分）：
   - `docs/设计文档.md`
   - `docs/架构文档.md`
   - `docs/依赖文档.md`
   - `.cursor/rules/mini-openclaw-project-constraints.mdc`
2. 在文档中完成以下工作（使用编辑工具修改，而不是只在对话中描述）：
   - **设计文档**：补充/修正对应条目的需求说明，保证：
     - 输入/输出示例清晰；
     - 与 P0/P1 分层描述一致；
     - 与现有条目编号、层级风格一致。
   - **架构文档**：补充对应模块的职责、依赖关系、调用链（例如新增 `command-exec` 模块，或者在 `chat` 模块下新增命令执行流程）。
   - **依赖文档**：如需求会引入新依赖或新增使用方式，需要在这里说明用途、范围和安全注意事项。
3. 文档更新完成后，用 2–3 句话概括你刚刚对文档做了哪些修改（不需要贴出完整 diff）。

### 4. Plan 阶段：规划架构与模块边界

> 当需求明显会影响多个模块或引入新子系统（例如命令执行、定时任务、MCP 调用）时，应主动考虑切换到 Plan 模式来先设计方案，再动代码。

1. 如果任务较大或跨多个模块：
   - 使用 `SwitchMode` 切换到 Plan 模式（需要时征求用户确认即可简短说明原因）；
   - 在 Plan 模式下给出一份**简洁的实现方案**，包括：
     - 将会修改/新增的模块和文件；
     - 每个模块的职责边界；
     - 关键数据结构与接口（例如内部 TypeScript 类型与函数签名）；
     - 与现有 CLI 命令、配置、存储的交互关系。
2. 即便不切换模式，也要在同一条回答中列出一个**有序步骤列表**（Step 1, Step 2, …），用来指导后续工具调用和代码修改。
3. 明确说明哪些属于 P0 主链路（必须实现），哪些属于 P1/加分项（有时间再做）。

### 5. 判断是否需要新增依赖

在开始写代码前，基于项目约束和依赖文档做**显式依赖决策**：

1. 首先检查本项目约束：
   - 运行时使用 **Bun + TypeScript**；
   - 已允许的核心依赖：`commander`、`@clack/prompts`、`chalk`、`openai`；
   - Bun 自带的能力（`fs`、`readline`、`fetch` 等）优先使用，不重复安装；
   - 不使用 `npm`/`pnpm`/`vite` 等其他工具链。
2. 分析本次需求是否可以在**现有依赖和标准库**内完成：
   - 如果可以，明确写出「**本次实现不新增依赖**」，并说明理由（例如仅用 `child_process`/Bun 的等价能力即可）。
   - 如果确实需要新增依赖：
     - 写清楚依赖名称、用途、预期使用范围（模块级，而不是全局乱用）；
     - 在 `docs/依赖文档.md` 中补充说明；
     - **不要实际运行安装命令**，只提示用户可在本地执行形如 `bun add xxx` 的命令（遵守用户「bun install 由用户手动执行」的约定）。

### 6. 代码实现（遵守分层与模块边界）

按计划步骤进行具体实现时，遵守以下原则：

1. **分层与目录约定**：
   - CLI 入口：例如 `src/cli.ts`。
   - 子命令层：`src/commands/*.ts`（如 `config.ts`、`chat.ts`）。
   - 职责模块层：`src/config/*`、`src/chat/*`、`src/memory/*`、`src/storage/*`、`src/llm/*`、以及必要的新模块（例如 `src/command-exec/*` 或类似命名）。
2. 使用编辑工具前，必须先用 Read 工具读取相关文件，以获取最新内容。
3. 修改代码时：
   - 优先保持现有风格与工程规则（包括 Biome 格式）；
   - 不写「解释性注释」来重复代码语义，只在设计特殊/有隐含约束处补充简短注释；
   - 对新增模块暴露清晰、窄接口，避免在 CLI 层直接操作底层细节。
4. 若本次需求有关**通过 chat 执行系统命令 / 定时任务 / MCP 调用**，在实现时要显式考虑：
   - 权限与安全边界（避免无约束执行任意命令）；
   - 如何在对话中展示输出（例如标准输出截断、错误信息格式化）；
   - 是否需要在记忆或日志中记录执行过的操作。

### 7. 测试与质量检查（质量门禁）

当你认为代码逻辑已实现完毕后，必须按本项目质量门禁执行检查流程：

1. 首先建议用户在本地运行：
   - `bun test`
   - `bun run lint`
   - `bun run format:check`
2. 如果你在本环境可以运行 Shell 工具，并且不违反用户显式约束，可以按以下顺序实际执行：
   - `bun test`
   - `bun run lint`
   - `bun run format:check`
3. 如果其中任何一步失败：
   - 优先尝试自动修复：
     - `bun run format`
     - `bun run lint:fix`
   - 再次运行：
     - `bun test`
     - `bun run lint`
     - `bun run format:check`
4. 如需手动修复，集中关注：
   - 新增/修改文件中的类型错误、未使用变量、导入路径错误；
   - 测试中与新功能直接相关的断言失败；
   - 不要在未修改过的旧代码上大规模重构，除非是为了解决测试或 linter 阻塞。

### 8. 对用户的最终交付格式

完成一次从设计文档到实现的闭环后，向用户给出的最终回答应简洁，包含：

1. **简要结果概述**（2–4 句）：
   - 这次实现了什么能力；
   - 主要改动集中在哪些模块/文件（只提模块名或简要路径）。
2. **文档更新说明**（1–2 句）：
   - 哪些文档章节被更新/新增。
3. **测试结论**：
   - 测试与质量检查是否全部通过；
   - 若未能全部通过，需要简要说明当前阻塞与建议的下一步（例如本地环境缺失、外部 API 不可用等）。

避免在最终总结中粘贴大量代码，只在必要时用少量代码引用指出关键实现点。
